# /doc-consolidate — 将对话沉淀为项目文档

> 关联：开发期 Agent Skill 定义见 `.claude/skills/doc-consolidator.md`（Phase 2 后 Agent 可自主调用）

请将当前会话中**最近的技术讨论**（从用户指定的起点开始，或从上一个文档化节点之后）整合到项目的文档体系中。

## 执行流程

### 第一步：扫描目标对话

找到用户指定的对话范围（默认为"从最近一次文档化之后的所有技术讨论"），提取所有**非闲聊**的内容段落。

### 第二步：分类

将每个内容段落归类到以下类别之一：

| 类别 | 判定标准 | 归档目标 |
|------|---------|---------|
| **架构决策** | 涉及技术选型、方案对比、"用还是不用"的结论 | `docs/decisions/` — ADR 格式 |
| **技术调研** | 对某个技术/工具/生态的系统性分析 | `docs/decisions/` 或 `.claude/plan/` — 调研补充 |
| **深层讨论** | 架构理解、核心概念澄清、设计思路推演 | `docs/dev-log/` — 开发日志 |
| **操作经验** | 踩坑记录、配置技巧、工具用法 | `docs/dev-log/` 或 `docs/claude-tips/` |
| **学习笔记** | 新技术的知识点、API 梳理 | `docs/learning-notes/` |
| **更新补充** | 内容补充到已有文档即可，不需要新文档 | 在现有文档中追加 |

### 第三步：检查去重

在创建任何文档之前，先 Globb 和 Grep 已有文档，确认：
- 是否存在覆盖相同主题的文档？
- 如果有，是更新现有文档还是新建？
- 新建文档的编号/命名是否与现有体系冲突？

### 第四步：按格式写出

#### ADR 格式（`docs/decisions/`）

```markdown
# ADR-XXX: 标题（决策关键词）

> 日期：YYYY-MM-DD · 状态：✅ 已采纳 / 🔄 讨论中 / ❌ 已否决

---

## 背景
（为什么要做这个决策？上下文是什么？）

## 决策
**一句话说清楚选了哪个方案。**

## 对比
| 维度 | 方案 A | 方案 B | 方案 C |
|------|:---:|:---:|:---:|
| ... | ... | ... | ... |

## 关键理由
1. **理由一**：...
2. **理由二**：...

## 替代方案（已否决/暂缓）
- **方案 X**：为什么没选
- **方案 Y**：什么情况下可以重新考虑

## 代价
- 选择了这个方案带来的负面成本

## 相关文档
- [相关 ADR](...) 
- [相关 dev-log](...)
```

#### 开发日志格式（`docs/dev-log/`）

```markdown
# YYYY-MM-DD — 标题（一句话概括）

> 可选引文（概括核心洞察）

---

## 讨论背景
（为什么要讨论这个？）

## 核心结论
### 子主题 1
...

### 子主题 2
...

## 关键洞察
- ...
- ...

## 相关文档
- [链接](...)
```

#### 更新现有文档

如果是补充更新，在目标文档的适当位置追加内容，并在 CLAUDE.md 的文档索引中更新引用。

### 第五步：编织引用网络

生成或更新以下交叉引用：

1. **新建文档内部**：在"相关文档"节链接所有相关文档
2. **被引用文档**：如果合适，在它们的"相关文档"节回链新文档
3. **CLAUDE.md**：在开发日志速览表或设计文档节添加新条目
4. **roadmap.md / architecture.md**：如果新文档与某个 Phase 或模块强相关，添加引用
5. **相关 dev-log / ADR**：在相关文档中追加引用链接

### 第六步：汇报

完成后的输出格式：

```
## 文档化完成

### 新建（N 篇）
| 文件 | 内容 |
|------|------|
| docs/decisions/008-xxx.md | ADR-008 — xxx |
| docs/dev-log/2026-xx-xx-xxx.md | xxx |

### 更新（M 篇）
| 文件 | 更新内容 |
|------|---------|
| CLAUDE.md | 新增 x 条引用 |
| roadmap.md | ... |
```

## 项目文档体系速查

| 目录 | 用途 | 编号/命名规则 |
|------|------|-------------|
| `docs/decisions/` | 架构决策记录 | `NNN-slug.md`（三位数递增） |
| `docs/dev-log/` | 开发日志 | `YYYY-MM-DD-slug.md` |
| `docs/claude-tips/` | Claude Code 技巧 | `slug.md` |
| `docs/learning-notes/` | 学习笔记 | `tech-slug.md` |
| `.claude/plan/` | 核心设计文档 | `architecture.md` / `roadmap.md` 等 |

## 现有文档清单（写前必查）

执行前必须 Globb `docs/**/*.md` 和 `.claude/plan/*.md`，确认最新状态，而不是依赖此列表。

---

## 注意事项

- **宁可少建文档，不要过度拆分**：一个话题如果可以从多个角度讨论，尽量合成一篇 dev-log 而不是拆成多个 ADR
- **ADR 数量节制**：不是每个讨论都值得一个 ADR。ADR 仅限于"选择了 A 而不是 B"的不可逆决策
- **日期用对话发生日期**：不是写文档的日期
- **写清楚"为什么"而不是"是什么"**：代码已经说了"是什么"，文档要补充"为什么这么做"
- **引用用相对路径**：方便项目整体移动
- **不重复 CLAUDE.md 已有内容**：如果 CLAUDE.md 已经记录，只追加引用，不重复写
