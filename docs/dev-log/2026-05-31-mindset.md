# 2026-05-31 — 开发思维转变：从"前端驱动"到"数据流驱动"

---

## 问题起源

作为前端开发背景，直觉反应是：

```
初始化 Nuxt → 写前端页面 → 接入数据库 → 开发后端 API
```

这个思路在 B 端表单/列表类项目中合理，但在这个 AI 对话项目中是错的。

---

## 核心洞察：UI 只占 10%

```
                    ┌─────────────────────────┐
                    │     Chat UI (壳)         │  ← 10% 的工作量
                    ├─────────────────────────┤
                    │   SSE 流式传输层         │
                    ├─────────────────────────┤
                    │   LLM Provider 层 (核心) │  ← 真正决定体验的地方
                    ├─────────────────────────┤
                    │   Database (对话元数据)   │
                    └─────────────────────────┘
```

项目的核心价值不在 UI，而在**流式对话的稳定性**和**模型切换的无缝性**。Nuxt UI v4 的 `ChatMessages` 组件一行代码就能渲染消息列表，但：

- LLM 流被 Cloudflare 100s 断开怎么办？
- Anthropic 和 OpenAI 的 SSE 格式不一样怎么统一？
- 工具调用的 JSON 怎么从流中完整提取？

这些才是真正花时间的事，而它们全部在服务端。

---

## 如果前端优先会怎样

写 `ChatPanel.vue` 的布局和动画 → 连不上后端，全是 mock → UI 调好了，写 API 时发现流式格式不兼容 → 回头改 composable → 最后发现最复杂的根本不是 UI

---

## 正确顺序：从"数据"往"界面"推

### 第一层：数据库（~30 分钟）

定义 Schema → `drizzle-kit push` → 确认 Neon 里表建好了。

**不需要任何 UI 代码。** 验证方式是打开 Neon Dashboard 看表结构。

### 第二层：LLM Provider（核心，最花时间）

三种 Provider 的流式格式统一为 `ReadableStream<string>`。

验证方式不是打开浏览器：

```bash
curl -N -X POST http://localhost:3000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"provider":"openai","model":"gpt-4o","messages":[{"role":"user","content":"你好"}]}'
```

**终端里看到流式输出文字 = 通过。**

### 第三层：SSE 心跳 + `/api/chat`

用 LLM Provider 的流 + 30s 心跳 + 正确响应头，包装成标准 SSE 端点。

**这个端点调通了，前端才有意义。**

### 第四层：对话 CRUD + 持久化

在 `/api/chat` 里加自动保存，再补列表/新建/删除。

### 第五层：Chat UI（最后）

此时所有 API 已可用，前端变成纯组装：
- `useChat()` → 调 `/api/chat`，消费 SSE 流
- `ChatMessages` → 渲染消息
- `ModelSelector` → 切换 provider
- `ConversationList` → 调 CRUD

---

## 两种顺序对比

| 维度 | 前端优先（旧思路） | 后端优先（新思路） |
|------|-------------------|-------------------|
| **第一个小时** | 调 `ChatPanel.vue` 布局 | 建表 + LLM Provider 接口定义 |
| **第一个可验证产出** | 静态 UI 截图 | `curl` 看到流式文字返回 |
| **风险发现时机** | UI 写完才发现流式格式问题 | 第一时间暴露格式兼容问题 |
| **UI 写完后** | 开始补后端，可能推翻 UI | 后端已稳定，UI 只是组装 |

---

## 一句话总结

> **把 Nuxt 项目当成一个后端项目来启动，前端只是最后一步的渲染层。**

心理上要换位：

- 先用 `curl` 验证，再用浏览器验证
- 先确保 `server/services/` 稳健，再写 `app/components/`
- 先在终端里看到流式输出，再考虑 UI 动画

数据流从 **LLM → Provider → SSE → API → Composable → 组件**，从源头往末端推，每一步都有可验证的产出。
