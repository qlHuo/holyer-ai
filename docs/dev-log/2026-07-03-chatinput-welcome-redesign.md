# 2026-07-03 — ChatInput 双区域重构 + 欢迎页快速操作设计

> 核心洞察：UI 控件选型中，"后续可能需要"不是换技术的充分理由，简单方案加扩展预留才是正确的工程判断。

---

## 讨论背景

Phase 1.5 剩余两项未完成任务：

- **1.18 ChatInput 优化**：原始计划是 contenteditable div 替代 textarea + 粘贴处理
- **1.24 页面初始化**：欢迎页 + 骨架屏优化

经过代码审查发现，当前 ChatPanel 已有一个极简欢迎视图（图标 + 问候语），但缺少"快速开始"的引导。ChatInput 仍是基础的单行 textarea 实现。本次讨论确定两个模块的最终方案。

---

## 决策一：保持 textarea，不做 contenteditable

### 结论
**否决 contenteditable div，继续使用原生 textarea。**

### 对比

| 维度 | textarea | contenteditable |
|------|----------|-----------------|
| v-model | 原生支持，双向绑定零代码 | 需手动同步 innerHTML，处理光标和选区 |
| placeholder | 原生 `placeholder` 属性 | CSS `:empty:before` 伪元素模拟 |
| 键盘行为 | Enter/Shift+Enter 跨浏览器一致 | 各浏览器换行行为不同，需逐一处理 |
| 粘贴 | 纯文本自动清洗 | 必须 `paste` 事件拦截 + HTML 解析 + 白名单过滤，否则富文本污染输入 |
| 光标管理 | 浏览器完全管理 | 大量边界 case（光标跑出编辑区、选区丢失、中文输入法兼容） |
| 富文本渲染 | 不支持 | **唯一优势**：输入区内代码高亮、链接渲染 |

### 关键理由

1. **当前场景是纯文本输入**，发送给 LLM 后由服务端返回 Markdown。输入区不需要富文本渲染。
2. **"后续可能需要 @ 提及"不构成理由**：@ 提及功能（如调用知识库、引用文档）完全可以用 textarea + 下拉弹窗实现——textarea 只存纯文本，弹窗做交互，不需要 contenteditable。Slack、Discord、GitHub 的 @ 提及都是 textarea + 浮层，不是 contenteditable。
3. **contenteditable 的维护成本是持续的**：每次浏览器更新都可能引入新的光标/选区 bug，中文输入法更是重灾区。以个人项目的维护投入，这不值得。
4. **DeepSeek、ChatGPT、Kimi 都用 textarea**，行业验证了这个选择。

### 替代方案（什么情况下可以重新考虑）

如果未来需要**输入区内 Markdown 实时预览**（如代码块语法高亮、粗斜体渲染），contenteditable 才有不可替代的价值。那时重新评估。

## 决策二：双区域布局 — 输入区 + 操作栏分离

### 结论
**ChatInput 拆为上下两个区域：上区是 textarea 卡片容器，下区是操作栏（模型选择 + 发送）。**

### 结构

```
重构前                              重构后
┌──────────────────────┐           ┌──────────────────────┐
│ [模型选择器]         │           │                      │
│ [textarea]    [发送] │           │  [textarea 卡片]      │
└──────────────────────┘           │  border + focus ring  │
                                   ├──────────────────────┤
                                   │ 🤖 模型选择    ✨ 发送 │
                                   └──────────────────────┘
```

### 关键理由

1. **视觉整体性**：textarea 外有卡片边框，输入控件成为一个"对象"，而非分散的零件。focus 时卡片边框高亮，视觉反馈更强。
2. **操作栏职责清晰**：模型选择是"发送前的配置"，应该在操作栏而非输入框上方（上方是"上下文"区域，如附件预览、Agent 状态）。
3. **扩展不破坏结构**：操作栏左侧是按钮组，未来加 Web 搜索开关、附件按钮、Agent 模式切换，直接往里插 `<button>`，不需要改动布局。
4. **参考 DeepSeek 的成熟设计**：双区域的视觉层次已经被大量用户验证。

### 操作栏不预留具体按钮

保持 `flex items-center justify-between` 结构即可。左侧 `gap-1` 按钮组，右侧发送按钮。不画占位符——没有功能的按钮比空位更干扰。

## 决策三：欢迎页不做独立页面，仅增强现有空态

### 结论
**不创建独立的欢迎页组件或路由。在 ChatPanel 现有的 `showWelcome` 区域增加 6 个快速操作卡片，点击填入输入框（不自动发送）。**

### 6 个快速操作卡片

| 图标 | 文案 |
|------|------|
| 📰 | 总结今天国内热点新闻 |
| 🤖 | 总结最新 AI 发展现状 |
| 💻 | 用 Python 写一个爬虫脚本 |
| 📝 | 帮我润色一段文字 |
| 🐛 | 帮我分析代码问题 |
| 🔍 | 解释一个技术概念 |

- 静态常量硬编码，后续换 API 只需替换数据源
- 桌面 3 列，移动端 2 列（CSS Grid 响应式）
- 点击 → 填入 ChatInput，用户可编辑后再发送

### 关键理由

1. **不区分"新对话"和"欢迎页"**：从产品视角看，空对话就是应该展示引导内容，创建对话和展示欢迎页是同一件事，不需要两套 UI。
2. **ChatInput 始终存在且复用**：无论欢迎态还是对话态，输入框只有一份。通过 `prefill` prop 从卡片传值到输入框，简单直接。
3. **成本极低**：不新增组件文件，直接在 ChatPanel 模板中加一个 grid 容器 + 6 个 card div。净增约 35 行。
4. **从"工具"变"产品"**：当前打开是空白输入框，加上引导卡片后第一印象完全不同。

### 为什么不做更多

- ❌ **不做分类**（创作/编程/知识）：6 个卡片不值得分类，分类增加认知负担
- ❌ **不做动态推荐**：需要后端 + 用户画像，投入产出严重倒挂，后续有需求时替换数据源即可
- ❌ **不做自动发送**：点击填入输入框但允许编辑，这是更尊重用户的交互模式

---

## 与其他模块的交互

```
ChatPanel
  ├── showWelcome = true → 渲染欢迎区域（问候 + 6 卡片）
  │                         prefillContent = suggestion.label
  └── showWelcome = false → 渲染消息列表

ChatInput (始终在 ChatPanel 底部)
  ├── props: { prefill: string }
  ├── watch(prefill) → input.value = prefill → emit('prefill-consumed')
  ├── 上区: textarea 卡片
  └── 下区: ChatModelSelector + 发送/停止按钮
```

`prefill` prop 是欢迎卡片和输入框之间的唯一通信通道。一个 ref + 一个 watch，不引入 store 或 event bus。

## 补充（2026-07-04）：统一卡片视觉优化 + nextTick 高度还原 Bug

> 核心洞察：双区域即使同在一个大容器中，如果 textarea 和工具栏各有自己的边框/背景，视觉上仍然是"两块"。真正的统一感来自**单个边框 + 单个焦点环 + 条件分隔线**。

### 问题：首次实现仍然"割裂"

按决策二的方案实现后（textarea 卡片 + 下方操作栏），用户通过 `/frontend-design` 反馈组件"割裂"——两个区域虽然都放在 `border-t` 容器中，但 textarea 有独立的 `border + rounded-xl`，操作栏是下方一条独立的 flex 行，视觉上是两块拼凑。

参考 DeepSeek 和千问的设计，它们把 textarea 和底部工具栏包在**同一个圆角边框卡片**内，聚焦时整张卡片高亮，形成"一个输入组件"的感知。

### 解决方案：合并为单卡片

```
之前（双区分离）                      之后（统一卡片）
┌─ border-t ─────────────────────┐    ┌─ 去掉 border-t ───────────────┐
│  ┌────────────────────────┐    │    │  ┌──────────────────────────┐  │
│  │  textarea (独立边框)    │    │    │  │  textarea (无边框)        │  │
│  └────────────────────────┘    │    │  │                          │  │
│                                │    │  │  ── 条件分隔线 ──        │  │
│  [模型选择]           [发送]   │    │  │  [模型选择]       [发送]  │  │
└────────────────────────────────┘    │  └──────────────────────────┘  │
                                      └────────────────────────────────┘
      两块独立视觉区域                    一个统一卡片，焦点环作用于整体
```

关键改动：

1. **合并为一个卡片**：textarea + 工具栏放入同一个 `border rounded-xl overflow-hidden` 容器，不再分开
2. **去掉外层 `border-t`**：统一卡片自己已有完整边框，外层只需 `px-4 py-3` 留白，不再画顶部分割线
3. **`focus-within` 整体高亮**：`focus-within:border-primary + shadow-sm` 作用于整张卡片，textarea 内部 `focus:outline-none`
4. **条件分隔线**：行数 ≥2 或有内容时才显示 textarea 和工具栏之间的分隔线，空态时干净无界，输入后自然分层
5. **间距微调**：textarea `pb-2`（紧凑衔接分隔线），工具栏 `pt-2 px-3 pb-3`（与卡片边缘留呼吸空间）

### `nextTick` 高度还原 Bug

**现象**：发送消息后 textarea 内容清空，但高度没有回到 1 行，直到再次输入时才正常。

**根因**：`handleSend()` 中 `input.value = ''` 后立即同步调用 `autoResize()`，此时 Vue 尚未更新 DOM，`el.scrollHeight` 仍反映清空前的多行高度，`el.rows` 不会被重置。

**修复**（`ChatInput.vue:55-59`）：

```ts
const content = input.value
input.value = ''
await nextTick()   // 等 Vue 更新 DOM，scrollHeight 反映空态
autoResize()       // 此时才能正确回到 1 行
await sendMessage(content)
```

这是一个经典的 Vue 响应式时序问题：修改 ref → DOM 更新是异步的，任何依赖 DOM 测量值的操作都必须在 `nextTick` 之后。预填充的 `watch` 里已经用了 `nextTick`，发送路径漏掉了。

### 其他微调

- **placeholder 更新**：从 "有什么可以帮助你的？" 改为 "有什么可以帮助你的？按 Enter 发送, Shift + Enter 换行"，降低新用户的学习成本
- **外层 `border-t` 移除**：统一卡片已有完整边框，外层容器不再需要顶部分割线
